    Background: Reason for Netfilter tables

    The lengthy discussion on implementing cluster networking and routing manually using iptables first, and then ipvs may provide some context, if required, for the kube-proxy nftable routing feature now generaly available in Kubernetes clusters. And this should be a concise introductory article with reference to dive deep resources.

    Motivation for Netfilter tables

    There is the initial and still default "iptables" modes, and the "ipvs" mode that adds more optimization to the iptables in large clusters. So why do we need yet a 3rd kube-proxy mode? Lets start by addressing this "why?" question. As in the articles linked above;

    • the iptables mode uses the default linux kernel iptables firewall filtering and routing. Pro, it is simple and well-known. Con, other than inbuilt non-optimal logic, etc. iptables rules can quickly grow large and unwieldy as the cluster size(service and endpoint) grows. While routing speed drastically reduces and overhead increase as rules grow, since every packet needs to be checked each rules till a match is foudn before it is routed.

    • ipvs mode then attempts to solve both the bottleneck of iptables' slow routing and large unwieldy ruleset as cluster size grows, with minimal virtual service sets, and servers. However, ipvs implementation, hashing, routing etc can be a bit non-intuitive. For example, it still uses some part of iptables rules creation to route cluster traffic, significantly lower number of rule required compared to just iptables, but it is still there. Also worth calling out, the intuitiveness of ipvs also introduced complexities in implementing kube-proxy cluster features with ipvs - sample issue 1, sample issue 2 ,sample issue 3, and so on

    • so nftables mode then attempts to remove dependence or tie-in with iptables mode, while offering the speed and robustness of the ipvs routing, and allowing the full suite of kube-proxy features to be available with more intuitive/simplified syntax and implementation. This feature availability is key. All this said, it is worth calling out that it may have no meaningful advantages to implement nftable mode on smal clusters. But in very large clusters, suggesting nftable mode may provide significant improvement and performance benefits for cluster networking.

    To implementation, again, we will do this manually to have some grasp of the plumbing of the traffic routing in the kernel as well a user-space(if available).

    Implementation

    So far so good, all these commands have done is set up a manual pod sandbox with simulated cni networking. No cluster service IP, node Port, etc. has been created. For service IP routing we use nftable to set up manually similar to how kube-proxy in nftables mode would.

    It may be worth starting by checking that you have nftables installed by default. On newer EKS optimized AMI from AL2023, thereabout, this should be the case. For older AMIs, you may want to install this. See this for some requirements documentation, but Linux kernels at or newer than 5.10, should meet the reuirements here.

    nft -a list ruleset && nft list tables
# if not present
yum install nftables -y
    1. lets start off by creating the nft table, called custom-proxy (play on the kube-proxy naming). In this table, we create a masquerading rule to allow hairpin traffic using cluster service IP. That is, a pod can connect to itself using its cluster service IP. (Notice that local loopback IP and pod loopback IP have been enabled in the cni steps above). Then next we create the custom pods endpoint rules, 1st one jumping to the masquerade chain for hairpin, and and build on this as we go.
    sudo nft add table ip custom-proxy

sudo nft add chain ip custom-proxy mark-for-masquerade
sudo nft add rule ip custom-proxy mark-for-masquerade "meta mark set meta mark | 0x00004000"

sudo nft add chain ip custom-proxy masquerading
sudo nft add rule ip custom-proxy masquerading "meta mark & 0x00004000 == 0x00000000 return"
sudo nft add rule ip custom-proxy masquerading "meta mark set meta mark ^ 0x00004000"
sudo nft add rule ip custom-proxy masquerading "masquerade fully-random"

CHAIN_1='endpoint-custoom-proxy-ns/pod-3/tcp/http__100.64.0.19/8080'
CHAIN_2='endpoint-custoom-proxy-ns/pod-4/tcp/http__100.64.0.20/8080'
sudo nft add chain ip custom-proxy "$CHAIN_1"
sudo nft add rule ip custom-proxy "$CHAIN_1" "ip saddr 100.64.0.19 jump mark-for-masquerade"
sudo nft add rule ip custom-proxy "$CHAIN_1" "meta l4proto tcp dnat to 100.64.0.19:8080"

sudo nft add chain ip custom-proxy "$CHAIN_2"
sudo nft add rule ip custom-proxy "$CHAIN_2" "ip saddr 100.64.0.20 jump mark-for-masquerade"
sudo nft add rule ip custom-proxy "$CHAIN_2" "meta l4proto tcp dnat to 100.64.0.20:8080"

    Note: keen readers may observe that for the service hairpin traffic above: pod-A -> service-A -> pod-A masquerading, SNAT-ing, the packet from the source pod is introduced, which is unlike the routing in iptables or ipvs mode either. This is because the initial connection goes through the conntrack table, and without SNAT, the reply packet attempts to go through the custom cni bridge back to itself, the originating pod-A. This causes an asymmetric routing which may be discarded as unexpected connection. To avoid this, we make sure traffic are symmetrically routed by enforcing the node IP as the source

    1. next, we build 1 single service / clusterIP chain and rules that expose pod-3 and pod-4 endpoints we just created in the custom-proxy table. This is how the cluster virtual services are implemented in the nft mode, and it also shows off the traffic/load balancing implementation of nftables from 1 service to 2 endpoints with a numgen random mod 2 ... random(statistically fair, but not workload fair, as established in prior articles) 50/50 distribution. There are fairer algorithms (advantages of ipvs as well) that you could play with in the nft rule to send connection to one endpoint or the other.
    SERVICE='service-custom-proxy-ns/custom-service/tcp'
sudo nft add chain ip custom-proxy "$SERVICE"
sudo nft add rule ip custom-proxy "$SERVICE" "numgen random mod 2 vmap { 0 : goto $CHAIN_A, 1 : goto $CHAIN_B }"
    1. lets create the routing for a generically selected service clusterIP value of 198.19.0.10, that send traffic to this endpoint to the $SERVICE chain created above.
    sudo nft add map ip custom-proxy custom-service-ips '{ type ipv4_addr . inet_proto . inet_service : verdict ; flags constant ; elements = { 198.19.0.10 . tcp . 8080 : goto service-custom-proxy-ns/custom-service/tcp } }'

sudo nft add chain custom-proxy custom-services
sudo nft add rule ip custom-proxy custom-services "ip daddr . meta l4proto . tcp dport vmap @custom-service-ips"
    1. the last building block to place now, is how to divert the flow of traffic through the cluster (worker nodes) to go through the nftable. We do this by creating base chains, can imagine inserting network hooks into certain point of the linux network flow. We use the destination prerouting for traffic to the pods, and source postrouting for traffic leaving the pods. These ensures the cluster service chain is looked at while deciding on routing
    sudo nft add chain ip custom-proxy nat-prerouting '{ type nat hook prerouting priority -100 ; }'
sudo nft add chain ip custom-proxy nat-postrouting '{ type nat hook postrouting priority 100 ; }'

# jump to services chain from the base chain nat-prerouting for pod traffic
sudo nft add rule ip custom-proxy nat-prerouting jump custom-services

# jump to masquerade chain from the base chain nat-postrouting for pod traffic
sudo nft add rule ip custom-proxy nat-postrouting jump masquerading

    And just like linkning lego blocks together, we have built out how Kubernetes cluster network routing works with the nftables mode. There are a few more intricacies that kube-proxy add for firewall, kube-VIP, dropped unsecure connections, etc. that we dont look into. We however, have a clear grasp of the main blocks that is used to set up successful routing; - chains for each pod endpoint(s) are created, and the enforcing node IP masquesrading for self-loopback using the pods' service IP - a chain for each service exposing pod endpoints are created, balancing traffic to the pod endpoint chain above - a map for cluster services to endpoint destination is created - and a chain sending service connection to the right element/entry in this map is also created - and lastly, there is a hook for the nftable into the Linc kernel network flow so this whole configuration is looked into while routing traffic.

    Test conncetion using cluster IP from one pod, or the other

    sudo ip netns exec veth-ns-test-pod-3 curl -ikv 198.19.0.10:8080

    Note here that while connection from the host to the pod works as we already established. Connection from the host to the service clusterIP, 198.19.0.10:8080, fails. Without getting bogged down with the details, the previous articles on iptables and ipvs describes how routing occurs, but the simple reason in local connection, the hook we setup in destination prerouting is completely bypassed. So we ca instead hook into the nat-output, similar to how kube-proxy would also set it up. Create the hooks and Test that connection as well.

    sudo nft add chain ip custom-proxy nat-output '{ type nat hook output priority -100 ; }'
sudo nft add rule ip custom-proxy nat-output jump custom-services

    Conclusion

    There is avenue to experiment further with our implementation by implementing a map for nodePorts, and in the service chain creating a rule that send nodePort service connection to the correct element in the nodePort map created. A few important further reading if testing this out, nodePort in iptables/ipvs mode that is more open, is more precise and cam only be reached using worker nodes' primary IP. Some 3rd party Kubernetes ingress controllers that broadcast traffic to all IP:NodePort may experience issue moving to nftables mode.

    Also, external connection fail from the pod when using cname instead of IP. This is because the dns traffic from the pod to a VPC .2 endpoint is not masked. And this traffic is dropped as unexpected in the VPC. You can look to send dns traffic through the mark-for-masquerading chain in the nftables.

    However, we wont be getting bogged down with implementing these features in this brief introduction into how this kube-proxy nftables mode works at this time. Again you can check out those previous articles for iptables and ipvs implementation for verbose discussions on routing.
